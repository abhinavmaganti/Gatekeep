Gatekeep

The inspection layer between autonomous AI agents and the hostile web.

Gatekeep turns untrusted pages into checked evidence before an agent can reason over them. It is not another chatbot. It is the checkpoint before the model.

Read the Case View System
The company case

AI agents are becoming browsers. Browsers need security.

Prompt injection is not a prompt problem. It is a boundary problem. Agents are being asked to read pages that can contain commands, traps, hidden text, and tool-hijacking instructions.

Gatekeep gives companies a place to enforce that boundary without asking the same model that might be attacked to also defend itself.

01

Before the model

Gatekeep sits upstream of the agent context. If content is malicious, it is never handed to the model as something to think with.

02

Separate judgment

The classifier is isolated from the working agent. A page can attack the source text, but it cannot inherit the agent's tools, memory, or task context.

03

Plain reasons

Every decision returns a reason a human can read: what was caught, why it matters, and what Gatekeep did with it.

04

Still useful

Gatekeep is not built to shut research down. It preserves safe evidence and removes the hostile instruction layer around it.

Operating model

The product story is simple: check, judge, watch.

Layer one

Catch the obvious attack.

Fast deterministic scanning for known prompt injection language, hidden text, encoded directives, and suspicious tool requests.

  • role override phrases
  • invisible or encoded payloads
  • exfiltration wording
Layer two

Ask a separate judge.

A model dedicated to one narrow job: decide whether the source is trying to hijack an assistant.

  • safe, suspicious, or malicious
  • dangerous spans cited
  • plain-English verdict
Layer three

Watch for drift.

Compare the agent's plan before and after reading approved content. Sudden tool calls and new goals get stopped.

  • before-plan capture
  • after-plan comparison
  • tool request review
